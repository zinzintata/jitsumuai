## 小売店の在庫担当者が棚卸後に直面するデータ突合の課題

小売店や中規模チェーンの在庫担当者にとって、毎月の棚卸は避けて通れない業務です。棚卸そのものの作業も負担ですが、それ以上に時間を取られるのが「棚卸後のデータ突合と差異分析」の工程です。

一般的な小売店の在庫管理担当者を想定すると、棚卸後の業務フローは次のようになります。

### 棚卸後に発生する典型的な作業手順

1. ハンディターミナルや紙の記録から、実在庫データをExcelに転記する
2. 基幹システムやPOSレジから理論在庫（帳簿在庫）データをCSVで出力する
3. 商品コード（SKU）をキーにして、VLOOKUP関数やIF関数で差異を算出する
4. 差異が一定以上の商品を目視で探し、リストアップする
5. 差異の原因を1件ずつ推測して分類する（入力ミス、盗難、破損、誤出荷、検品漏れなど）
6. 在庫が不足している商品を見極め、発注優先度リストを手作業で作成する
7. 上長やエリアマネージャー向けの報告書をまとめる

この一連の流れを1人の担当者が担っているケースは少なくありません。SKU数が数百から数千に及ぶ店舗では、差異の突合だけでも相当な時間がかかります。一般的な中規模小売店を仮定した場合、棚卸後のデータ突合と差異分析に3〜5時間程度を要することも珍しくないとされています。

さらに問題なのは、差異原因の分類が属人的になりやすい点です。「この商品カテゴリは万引きが多い」「この仕入先は検品漏れが起きやすい」といった判断はベテラン担当者の経験に依存しがちで、担当者が異動・退職すると引き継ぎが難しくなります。

## AIエージェントで棚卸差異分析を効率化する全体像

### このエージェントが担う3つの役割

本記事で紹介するAIエージェントは、棚卸差異データを入力として受け取り、以下の3つの処理を行います。

1. **差異の大きい商品の抽出**：設定した閾値（例：数量差5個以上、差異率10%以上）をもとに、注意が必要な商品を自動でリストアップします
2. **差異原因の仮分類**：差異のパターン（プラス差異・マイナス差異、差異の大きさ、商品カテゴリの傾向）から、原因カテゴリを仮分類します
3. **発注優先度リストの生成**：実在庫の水準と差異の状況を踏まえて、発注の優先度を高・中・低の3段階で整理します

ここで強調しておきたいのは、このエージェントが出す「原因分類」はあくまで「仮分類」であるという点です。最終的な原因の特定と判断は、現場を知る人間が行う必要があります。AIが担うのは、大量のデータの中から注目すべきポイントを絞り込む「下ごしらえ」の部分です。

### エージェントの処理フロー全体像

```
入力：棚卸差異データ（CSV/テキスト形式）
  │  商品コード、商品名、理論在庫数、実在庫数
  ↓
Step 1：差異算出と異常値の抽出
  │  差異数・差異率を計算し、閾値超えの商品をリストアップ
  ↓
Step 2：差異原因の仮分類
  │  差異パターンと商品カテゴリから原因カテゴリを推定
  ↓
Step 3：発注優先度の判定
  │  実在庫水準と差異傾向から優先度を3段階で評価
  ↓
出力：差異サマリー・要注意商品リスト・原因別分類・発注優先度リスト
```

LLM（大規模言語モデル）を活用したこのエージェントは、特定のプログラミング言語やAPIの構築を必要としません。後述する実装プロンプトをコピーして、お使いのLLMサービスに貼り付けるだけで試すことができます。

## Before/After：手作業Excelフロー vs AI活用後のフロー比較

手作業による従来のフローと、AIエージェントを活用した場合のフローを比較します。

| 項目 | Before（導入前） | After（導入後） |
|---|---|---|
| データ突合 | Excelで商品コードをキーにVLOOKUP/IF関数で手作業突合 | CSVデータをAIに入力し、差異を自動算出 |
| 異常値抽出 | 差異列を目視で確認し、大きい値を手動でピックアップ | 閾値を設定し、該当商品を自動抽出・ソート |
| 原因分類 | 担当者が1件ずつ経験則で推測・記入 | AIが差異パターンから仮分類を提示、担当者が確認・修正 |
| 発注優先度の判断 | 在庫状況と売れ行きを頭の中で照らし合わせて判断 | AIが在庫水準と差異傾向から優先度案を生成 |
| 関与する人員 | 担当者1名が全工程を手作業で実施 | 担当者は確認・修正・承認に集中 |
| 作業手順 | 7〜8ステップ（転記・関数設定・目視確認・分類・リスト作成） | 4〜5ステップ（データ出力・AI入力・結果確認・修正・報告） |
| 発生しやすい問題 | 転記ミス、関数設定の誤り、属人化した原因判断 | AIの仮分類の誤り（人間の確認で対応） |
| 工数（目安） | 1回の棚卸あたり3〜5時間程度（一般的な中規模店舗を仮定） | 条件によっては1〜2時間程度に短縮できる可能性あり |

> **注記**：上記の比較は、SKU数500〜2,000程度の中規模小売店を一般的な例として想定しています。実際の効果は商品数、データ品質、担当者のスキル、使用するLLMサービスによって大きく異なります。

Afterフローのポイントは、担当者の役割が「作業者」から「確認者・判断者」に変わることです。データの転記や関数の設定といった機械的な作業をAIに任せ、担当者は原因分類の妥当性確認や最終的な発注判断に集中できるようになります。

## 棚卸差異チェックAIエージェントの処理フロー詳細（3ステップ構成）

エージェントが実行する3つのステップを、それぞれ詳しく解説します。

### Step 1：差異データの読み込みと異常値抽出

最初のステップでは、入力された棚卸差異データ（理論在庫数と実在庫数）をもとに、以下の計算を行います。

- **差異数**：理論在庫数 - 実在庫数（プラスなら在庫不足方向、マイナスなら在庫過多方向）
- **差異率**：差異数 / 理論在庫数 x 100（%）
- **絶対値での差異の大きさ**：差異数の絶対値

計算後、設定された閾値（デフォルトでは数量差5個以上、または差異率10%以上）を超える商品を抽出します。抽出された商品は差異の大きい順にソートされ、「要注意商品リスト」として出力されます。

### Step 2：差異原因の仮分類

抽出された要注意商品に対して、差異のパターンから原因カテゴリを仮分類します。分類に使う主な判断基準は以下のとおりです。

| 差異パターン | 仮分類される原因カテゴリ |
|---|---|
| マイナス差異（実在庫 < 理論在庫）が小さい | 入力ミス・計上漏れの可能性 |
| マイナス差異が大きい（特に高額商品） | 盗難・紛失の可能性 |
| マイナス差異が特定カテゴリに集中 | 検品漏れ・誤出荷の可能性 |
| プラス差異（実在庫 > 理論在庫） | 返品未計上・入荷二重計上の可能性 |
| 差異が周期的に発生 | システム連携のエラー・タイムラグの可能性 |

この分類はあくまでパターンに基づく「仮説」です。AIは現場の状況を知らないため、実際の原因は担当者が確認する必要があります。仮分類の横に「確認ステータス」欄を設け、担当者が「確認済み」「原因修正」等を記入できる構成にしています。

### Step 3：発注優先度の判定とリスト生成

最後のステップでは、差異分析の結果を踏まえて、発注の優先度を3段階で判定します。

- **高（urgent）**：実在庫がゼロまたは極端に少なく、欠品リスクが高い商品
- **中（normal）**：実在庫が一定数あるが、差異が大きく在庫水準に不安がある商品
- **低（watch）**：差異はあるが実在庫に余裕があり、次回発注時に調整すれば対応可能な商品

発注優先度の判定には、実在庫数と差異の方向（不足方向か過多方向か）を主な判断材料として使います。重点管理商品（売れ筋商品や季節商品など）をあらかじめ指定しておくと、それらの商品の優先度を引き上げる処理も行います。

# 制約事項（Constraints）
- 原因分類はあくまで「仮分類」であり、「確定」「断定」の表現は使わない
- 「盗難」と断定せず「盗難・紛失の可能性」と表現する
- 入力データに含まれていない情報（売上データ、季節性、過去の差異傾向等）は推測で補わず、「確認が必要です：[項目名]」と記載すること
- 発注数量の最終判断は必ず人間の担当者が行うことを前提とする
- レポートの末尾に「本レポートはAIによる仮分析です。最終判断は担当者が行ってください」と記載すること
- 最終出力は必ず人間が確認・承認してから使用すること

## カスタマイズガイド：自店の商品カテゴリ・発注ルールに合わせる方法

上記の実装プロンプトは汎用的に設計されていますが、自店の事情に合わせてカスタマイズすることで精度が向上します。ここでは主なカスタマイズポイントを紹介します。

### 閾値の調整

差異閾値は店舗の規模や商品特性に合わせて調整してください。

- **少量多品種の店舗**（雑貨店、文具店など）：数量差3個以上、差異率15%以上が目安
- **大量在庫の店舗**（ドラッグストア、ホームセンターなど）：数量差10個以上、差異率5%以上が目安
- **高単価商品を扱う店舗**：数量差1個以上（高額商品は1個の差異でも注目すべき）

### 原因分類ルールの追加

プロンプトのStep 3にある原因分類ルールは、一般的なパターンに基づいています。自店で特有の差異原因がある場合は、ルールを追加するとよいでしょう。

たとえば、生鮮食品を扱う店舗であれば「マイナス差異かつカテゴリが生鮮 → 廃棄処理漏れの可能性」というルールを追加できます。

### 発注ロット単位の設定

仕入先ごとに発注ロット単位が異なる場合は、「ケース単位（12個入り）」「パレット単位（48ケース）」のように具体的に指定してください。ロット単位を設定すると、推奨発注数がロット単位に切り上げて出力されます。

### 重点管理商品の指定

売れ筋商品、季節商品、利益率の高い商品など、欠品を特に避けたい商品の商品コードを重点管理商品リストに追加してください。これにより、該当商品の発注優先度が自動的に1段階引き上げられます。

## 発展的な使い方：定期棚卸データの時系列比較と傾向把握

本記事で紹介したエージェントは、1回分の棚卸データの分析を想定しています。継続的に活用する場合は、以下のような発展的な使い方も考えられます。

### 複数月分のデータを連続で分析する

毎月の棚卸後にこのエージェントを使い、出力されたレポートを保存しておきます。数か月分のレポートが蓄積されたら、「過去3か月分のレポートを比較して、差異が繰り返し発生している商品を抽出してください」という追加の指示をLLMに与えることで、傾向分析が可能です。

### 季節変動との突合

特定の商品が特定の時期に差異が増える傾向がないか確認することで、棚卸のタイミングや管理方法の改善につなげられます。たとえば「夏場に飲料カテゴリのマイナス差異が増える」という傾向が見えれば、検品体制の強化や棚卸頻度の見直しといった対策を検討できます。

### 店舗間の比較

複数店舗を管理するエリアマネージャーの場合、各店舗の差異分析レポートを比較することで、特定店舗に差異が集中していないかを確認できます。この場合、入力データに「店舗名」列を追加し、店舗ごとの集計を指示するプロンプトの拡張が必要です。

## 導入時の注意点・限界：データ品質が結果を左右する

このAIエージェントを活用する際に知っておいていただきたい注意点と限界をまとめます。

### 向いているケース

- SKU数が100以上あり、手作業での差異チェックに時間がかかっている場合
- 棚卸差異データがCSVやテキスト形式で出力できる環境がある場合
- 差異分析の属人化を解消し、手順を標準化したい場合
- 差異の「当たり付け」を素早く行い、確認・対応に時間を集中したい場合

### 向いていないケース

- **SKU数が極端に少ない場合（数十件以下）**：手作業で十分対応でき、AIを介するメリットが薄い可能性があります
- **データの電子化ができていない場合**：手書きの棚卸表のみで運用している場合、まずデータの電子化が先決です
- **差異原因の確定を自動化したい場合**：本エージェントの原因分類は「仮分類」であり、原因の確定は人間の判断が必要です。現場状況、防犯カメラの映像、取引先との照合など、AIでは取得できない情報が原因特定には不可欠です
- **リアルタイムの在庫管理を求める場合**：本エージェントは棚卸後のバッチ処理を想定しています。リアルタイムの在庫監視には、専用の在庫管理システム（WMS等）の導入を検討してください

### よくある失敗パターンと対処法

| 失敗パターン | 原因 | 対処法 |
|---|---|---|
| 差異データがうまく読み込まれない | CSVのカラム名が想定と異なる | プロンプトの入力仕様に合わせてカラム名を統一するか、プロンプト内のカラム名を修正する |
| 原因分類が実態と合わない | 分類ルールが自店の状況に適合していない | Step 3の分類ルールを自店の実績に基づいて調整する |
| 発注優先度が高い商品が多すぎる | 閾値の設定が厳しすぎる | 差異閾値を引き上げるか、発注優先度の判定基準（実在庫の割合）を調整する |
| AIの出力が長すぎて読みにくい | SKU数が多く、全件出力されている | 閾値を引き上げて抽出対象を絞るか、「上位20件に限定して出力してください」と指示を追加する |

### データ品質に関する注意

AIエージェントの分析精度は、入力データの品質に直接依存します。具体的には以下の点にご注意ください。

- 棚卸時のカウントミスがそのまま差異として検出されます。AIはカウントミスと実際の差異を区別できません
- 理論在庫データが最新でない場合（棚卸直前の売上・入荷が未反映の場合）、見かけ上の差異が大きくなります
- 商品コードの表記ゆれ（全角・半角の混在、ハイフンの有無など）があると、正しく突合できない場合があります

## まとめ：次回棚卸から始める差異分析の効率化ステップ

本記事では、小売店の在庫担当者が棚卸後に行う差異分析業務を、AIエージェントで効率化する方法を紹介しました。

ポイントを整理します。

- 棚卸差異データ（CSV/テキスト）をAIエージェントに入力すると、差異の大きい商品の抽出、原因の仮分類、発注優先度リストの生成を自動で行える
- AIの出力はあくまで「仮分類」「参考情報」であり、最終判断は担当者が行う前提で設計されている
- 閾値、原因分類ルール、重点管理商品リストを自店に合わせてカスタマイズすることで、分析精度を高められる
- データ品質（カウント精度、理論在庫の鮮度、商品コードの表記統一）が結果を左右するため、入力データの整備が重要

**次回の棚卸から試せるアクション：**

1. 今回の棚卸データをCSV形式で出力し、本記事の実装プロンプトをコピーしてLLMに貼り付けてみる
2. 出力結果と、自分が手作業で行った分析結果を比較し、AIの仮分類の精度を確認する
3. 差異が合っている部分・合っていない部分を見極め、原因分類ルールを自店向けに調整する

最初から完全に業務を置き換えるのではなく、「手作業の分析結果と照らし合わせて検証する」ところから始めることをお勧めします。数回の棚卸サイクルを経て、信頼できる精度が確認できた段階で、段階的に業務に組み込んでいくのが現実的な進め方です。
